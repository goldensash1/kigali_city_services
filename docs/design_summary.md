# Design Summary (1–2 pages)

## 1. Firestore Data Structure

The backend is organized into two primary collections:

### `users` collection
- **Document ID:** Firebase Auth UID
- **Fields:**
  - `uid` (String)
  - `email` (String)
  - `displayName` (String?)
  - `emailVerified` (bool)
  - `createdAt` (Timestamp)

### `listings` collection
- **Document ID:** auto-generated by Firestore
- **Fields:**
  - `name` (String)
  - `category` (String)
  - `address` (String)
  - `contactNumber` (String)
  - `description` (String)
  - `latitude` (double)
  - `longitude` (double)
  - `createdBy` (String, UID reference)
  - `createdAt` (Timestamp)

This structure supports global browsing, ownership-based updates/deletions, and map rendering from persisted coordinates.

## 2. Listing Model and CRUD Strategy

Listings are represented by a dedicated model (`Listing`) with:
- Firestore mapping (`fromFirestore`, `toMap`)
- immutable copy updates (`copyWith`) for edit workflows

CRUD operations are implemented in `ListingService`:
- `createListing`
- `watchAllListings`
- `watchListingsByUser`
- `updateListing`
- `deleteListing`

The UI never calls Firestore directly; widgets invoke controller methods and consume providers.

## 3. State Management Design (Riverpod)

Riverpod is used to connect data layer to UI layer:

- **Service providers:** Firebase Auth and Firestore services
- **Stream providers:** auth state, user profile, all listings, my listings
- **State providers:** search query, selected category, notification toggle
- **Controller providers:** mutation actions and loading/error state for auth/listing operations

`AsyncValue` is used consistently to render loading indicators and errors across screens.

## 4. Navigation and Screen Composition

The app uses a bottom navigation shell with four required screens:
1. Directory
2. My Listings
3. Map View
4. Settings

From Directory/My Listings, users can open Listing Detail. My Listings additionally provides edit/delete actions and add-listing entry.

## 5. Search, Filtering, and Real-Time Updates

Search and category filtering are implemented with provider-composed state:
- query text (`searchQueryProvider`)
- selected category (`selectedCategoryProvider`)
- filtered result (`filteredListingsProvider`)

Because listing data comes from Firestore streams, filtered results update dynamically as backend data changes.

## 6. Map and Navigation Integration

Two map experiences are implemented:
- **Map View screen:** all listings as markers
- **Detail screen:** focused marker for selected listing

Navigation button launches external Google Maps directions URL using the selected listing coordinates.

## 7. Key Trade-offs

- Used local state provider for notification preference (assignment allows local simulation)
- Chose Riverpod over Bloc to reduce boilerplate and keep async stream handling concise
- Used text-based lat/lng entry for MVP speed and explicit coordinate control
